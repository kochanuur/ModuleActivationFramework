<a id=1-概要></a>
## 1. 概要

<a id=1-1-目的></a>
### 1-1. 目的
- 本書は、独自フレームワーク `Module Activation Framework (以降 MoAF)` の仕様書である。
- `MoAF` の概要・提供機能および想定される使用方法について記載する。

<a id=1-2-moaf-とは></a>
### 1-2. `MoAF` とは
- `MoAF` は、プラグイン機能の提供と、コンテナを用いたインスタンス管理によるDI (Dependency Injection : 依存性注入) の提供を同時に行う独自フレームワークである。
- `MoAF` を使用することで、モジュール間を疎結合にし、かつホストアプリケーションの再コンパイルを行わずにモジュールを追加/削除することが可能になる (プラグイン機能) 。
- 本書更新時点で、WindowsFormアプリケーション (.Net) に特化している。
- WPFのフレームワークである `Prism` を参考に作成されている。

<a id=2-用語定義></a>
## 2. 用語定義
| 用語 | 意味 |
|-----|------|
| モジュール | `MoAF` を使用可能なクラス<br>プラグイン対象 |
| カタログ | モジュール候補を列挙する仕組み |
| コンテナ | 依存性注入を行うためのDIコンテナ |
| ライフサイクル | モジュールの状態遷移 (初期化中 → 初期化完了 など) |
| ライフスパン | モジュールの寿命 |
| Singleton | アプリケーション内で各インスタンスが1つしか生存できない性質 |
| Transient | 要求されるたびにインスタンスを生成する性質 |

<a id=2-1-モジュール></a>
### 2-1. モジュール
- コンテナへの登録、コンテナからの取得を可能にするインターフェース (以降、`MoAFインターフェース`) を継承しているクラスが該当する。

<a id=2-2-カタログ></a>
### 2-2. カタログ
- ディレクトリに存在する全DLLを確認し、モジュール候補を自動検出する。

<a id=2-3-コンテナ></a>
### 2-3. コンテナ
- モジュールおよび、非モジュール (`MoAFインターフェース` を継承していないクラス) の登録、取得制御を行う。 (将来的には削除制御も導入予定)

<a id=2-4-ライフサイクル></a>
### 2-4. ライフサイクル
- ライフサイクルは以下の7通り。
  - モジュール登録開始前
  - モジュール読み込み中
  - モジュール読み込み完了
  - モジュール内部設定中
  - モジュール内部設定完了
  - モジュール初期化中
  - モジュール初期化完了

<a id=2-5-ライフスパン></a>
### 2-5. ライフスパン
- 現状は `Singleton` のみ設定可能。将来的に `Transient` を導入予定。

<a id=2-6-singleton></a>
### 2-6. Singleton
- アプリケーション内で、各クラスにつきインスタンスが1つしか生成できないため、寿命はアプリケーション終了 (厳密にはコンテナ終了) までとなる。
- `MoAF` では、コンテナ登録時にのみインスタンスを生成する。

<a id=2-7-transient-未実装></a>
### 2-7. Transient (未実装)
- 要求されるたびにインスタンスを生成するため、寿命はその場限りとなる。
- `MoAF` では、取得時にインスタンスを生成する予定。

<a id=3-moaf-の提供機能></a>
## 3. `MoAF` の提供機能

<a id=3-1-moaf-が提供する機能></a>
### 3-1. `MoAF` が提供する機能
- 指定ディレクトリ内のモジュール自動検出 (プラグイン機能)
- DIコンテナを用いたモジュール管理 (DI機能)

<a id=3-2-機能詳細></a>
### 3-2. 機能詳細
1. ホストアプリケーションは、カタログ処理を実施するディレクトリを指定する。
2. `MoAF` は、指定されたディレクトリに対しカタログ処理を実施する。 -> ライフサイクル : モジュール登録開始前
3. 2.で検出されたモジュールを `Singleton` でコンテナに登録する。 -> ライフサイクル : モジュール読み込み中 ~ 完了
4. 3.で登録したモジュールに対し、モジュール内部のコンテナ登録を行う。 -> ライフサイクル : モジュール内部設定中 ~ 完了
5. 3.で登録したモジュールに対し、初期化処理を行う。 (モジュール側でコンテナからインスタンス取り出しも可能) → ライフサイクル : 初期化中
6. ホストアプリケーションに、3.で登録したモジュールの初期化が完了したことを通知する。 -> ライフサイクル : 初期化完了
7. 3.~6.をカタログ処理で取得した全モジュールに対して実施する。
8. 実施完了後、全モジュールのライフサイクルが `初期化完了` であることを確認し、全モジュールの初期化が完了したことをホストアプリケーションに通知する。

<a id=4-コンテナ></a>
## 4. コンテナ

<a id=4-1-コンテナ構造></a>
### 4-1. コンテナ構造
- コンテナは、1つのインスタンスに対し以下の情報群を所持している。
  - ライフスパン
  - サービスタイプ
  - 実装タイプ
  - 登録名
  - インスタンス
  - インスタンス作成状況

<a id=4-1-1-ライフスパン></a>
#### 4-1-1. ライフスパン
- 登録するインスタンスのライフスパン。
- 現状は `Singleton` のみ。

<a id=4-1-2-サービスタイプ></a>
#### 4-1-2. サービスタイプ
- 登録するインスタンスのクラス、または継承しているインターフェース/クラス。
- ユーザーが明確に指定しない場合、登録するインスタンスのクラスが入る。

<a id=4-1-3-実装タイプ></a>
#### 4-1-3. 実装タイプ
- 登録するインスタンスのクラス。

<a id=4-1-4-登録名></a>
#### 4-1-4. 登録名
- 取得の際に使用可能な検索キー。
- 設定は任意のため、未指定も可。

<a id=4-1-5-インスタンス></a>
#### 4-1-5. インスタンス
- 作成されたインスタンス。

<a id=4-1-6-インスタンス作成状況></a>
#### 4-1-6. インスタンス作成状況
- 以下の2つの状態を持つ。
  - 未作成
  - 作成済み

<a id=4-2-コンテナへの登録></a>
### 4-2. コンテナへの登録

<a id=4-2-1-singleton-登録></a>
#### 4-2-1. `Singleton` 登録
1. サービスタイプ、実装タイプ、登録名が全て一致した場合はリストに対し新規登録を行わない。 (登録内容が重複するため)
2. 実装タイプが同じで、サービスタイプまたは登録名が異なる場合は、同じインスタンスを指定してリストへ登録する。
3. 実装タイプが異なる場合は、インスタンスを新規作成しリストへ登録する。

<a id=4-2-2-transient-登録></a>
#### 4-2-2. `Transient` 登録
- T.B.D.

<a id=4-3-コンテナからの取得></a>
### 4-3. コンテナからの取得
- 取得にあたり、ユーザーは以下の2つの情報を指定する。
  - 取得タイプ
  - 検索キー

<a id=4-3-1-singleton-取得></a>
#### 4-3-1. `Singleton` 取得
1. 取得タイプと実装タイプが一致するインスタンスが存在する場合。
    1. 検索キーと登録名が一致するものを取得する。
    2. 1.が満たされなかった場合、かつ取得タイプと実装タイプの一致が1つのみの場合はそれを取得する。
    3. 1.と2.を両方満たさなかった場合は取得不能。
2. 取得タイプと実装タイプが一致するインスタンスがなく、取得タイプとサービスタイプが一致するインスタンスが存在する場合。
    1. 1-1.と同様。
    2. 1-2.と同様。
    3. 1-3.と同様。

<a id=5-各モジュールにおけるコンテナ登録取得></a>
## 5. 各モジュールにおけるコンテナ登録・取得
- 各モジュールでは、モジュール内部設定処理にてコンテナへの登録、モジュール初期化処理にてコンテナからの取得を行うことができる。
- コンテナへの登録については、モジュール内で使用しているクラスの登録が可能である。
  例を以下に示す。
  - とあるモジュールに、`Manager` クラスと `Calculator` クラスがあったとする。
  - `Manager` クラスは `Calculator` クラスを所有しており、コンストラクタで `Calculator` クラスを受け取るとする。
  - モジュール内部設定処理で、`Calculator` クラスをコンテナ登録後に、`Manager` クラスを登録する。
  - こうすることで、自身でインスタンス化を一切せずに、`Manager` クラスは `Calculator` クラスを所有可能になる。
  - 省略したが、`Calculator` クラスが `ICalculator` インターフェースを継承していた場合、`Manager` クラスにおいてDIが可能になる。
- コンテナからの取得については、原則自身のモジュールで登録したインスタンスのみ取得可能とする。
  - 他モジュールにおける登録名を知っていれば取得は可能だが、それを行うとモジュール間の結合が強まるため、原則は禁止。
  - しかし、システム的には可能なため、ユーザーの裁量次第である。

<a id=6-想定する使用方法></a>
## 6. 想定する使用方法
- `MoAF` は、以下のように使用されることを想定している。

<a id= 6-1-ユーザ定義の共通インラーフェースについて></a>
### 6-1. ユーザー定義の共通インターフェースについて
- 各モジュールは、ユーザー定義の共通インターフェースを継承する。
- 共通インターフェースは、モジュールが`実装しているインターフェース`と、モジュールが`使用するインターフェース`をそれぞれ取得するための受け口を用意する。
  - 例えば、`Calculator` モジュールは、`ICalculator` インタフェースを実装 (継承) し、`IAdd` インターフェースを使用するなど

<a id= 6-2-各モジュールについて></a>
### 6-2. 各モジュールについて
- 各モジュールは、他モジュールとやりとりを行うインターフェースとして機能する。
- `MoAFインターフェース` と、ユーザー定義の共通インターフェースの他に、ユーザー定義のインターフェースを最低1つは継承している。 (継承していない場合、ホストアプリケーションや他モジュールからの参照が不能)

<a id= 6-3-ユーザのモジュール管理について></a>
### 6-3. ユーザーのモジュール管理について
- ユーザー側では、`MoAF` から受け取った各モジュールにおいて、共通インターフェースを継承しているもののみを抽出する。 (全モジュールが該当するはず)
- 抽出した各モジュールについて、共通インターフェースを使用し、`実装しているインターフェース`と、`使用するインターフェース`を取得する。
- `使用するインターフェース`を実装している全モジュールを渡す。
- こうすることで、各モジュールは他モジュールの実体を知ることなく、インターフェースを経由してやりとりすることができる。
- ホストアプリケーションが既知のインターフェースを使用すれば、ホストアプリケーションの再コンパイルなしに、モジュールの追加が可能となる。
